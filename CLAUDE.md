# Notes for AI assistants and new developers

## ⚠️ Do NOT read the large data files into context

- **`data/SP500_history.csv` (~2.5 MB, 606 data rows).** Each row is a membership-change date plus the full ~500-ticker list as a python list literal (`['A', 'AAPL', ...]`). Reading it wholesale wastes enormous amounts of tokens and tells you nothing the schema doesn't. If you need to inspect it, use shell commands instead, e.g.:
  ```bash
  head -c 300 data/SP500_history.csv                    # header: date,tickers; list-literal format
  cut -d, -f1 data/SP500_history.csv | sed -n '2p;$p'   # date range: 1996-01-01 .. 2026-07-11
  ```
- **`data/cache/`** (gitignored, generated at runtime) holds everything fetched from Sharadar — `adj_close_cache.csv` can grow to tens of MB; same rule applies.
- `Precision_Recall_Tradeoff.csv` (~100 lines) is small enough to read if needed.

## Project map

- `pickn.py` — main study pipeline (`run_top_n_study`), yearly-label simulation (`simulate_year_model_selection`), grid sweep (`sweep_recall_precision_pairs`), criterion feasibility screen (`screen_criteria_feasibility`), plotting, and the Sharadar fetch/cache layer (`fetch_adj_close`/`download_adj_close`). Entry point: `uv run python pickn.py` — an argparse CLI with `study` (default when no subcommand is given), `sweep`, and `screen` subcommands; see `--help` on each.
- `simulate_model.py` — pure combinatorics/sampling, no network or market data. `simulate_selection` builds a random selection hitting a target (recall, precision); `estimate_num_ways` counts how many such selections exist; `sample_confusion_draws` bulk-draws many such selections at once (vectorized index arrays). `pickn.py` imports from here; nothing imports `pickn.py`.
- `constituents.py` — legacy standalone CLI for fetching point-in-time S&P 500 membership from Wikipedia revision history. Not imported by the other modules; `pickn.py` reads the committed `data/SP500_history.csv` instead (different format, Sharadar-native tickers).
- `docs/sharadar_docs.md` — how the Sharadar/Nasdaq Data Link API is used (tables, filters, pagination, adjustment methodology).

## Conventions and gotchas

- Tickers are **Sharadar-native and used verbatim**: share classes keep their dot (`BRK.B`, `BF.B`) and numeric suffixes (`ABI1`) are Sharadar's disambiguation of reused symbols. The old yfinance `.` → `-` normalization is gone — do not reintroduce it; `data/SP500_history.csv` and SHARADAR/SEP share the same namespace.
- The "positive" class label is the string `"TRUE"` (not a boolean) — see `positive_label` parameters.
- `get_sp500_tickers_by_year` returns membership **as of January 1** of each year: the file only records change dates, so year Y's snapshot is the last row dated on or before Y-01-01 (which can be a prior-year row). Intra-year index changes are still ignored. Years before the file's first row get no entry.
- Prices are Sharadar `closeadj` (split+dividend backward-adjusted). Equities come from `SHARADAR/SEP`; tickers passed via `fund_tickers` (the benchmark, e.g. SPY) that SEP has nothing for are retried against `SHARADAR/SFP`. Only *declared* funds get the SFP retry — a delisted stock's symbol can be reused by a fund, so a blind fallback could splice in the wrong security. The API key is read from `~/.nasdaq/data_link_apikey` by the `nasdaqdatalink` client automatically; requests are batched (`SHARADAR_TICKER_BATCH`) to stay under the client's pagination cap.
- Survivorship bias: per-year stock returns go through `year_universe_returns` (`pickn.py`), which keeps mid-year delistings (return measured to last available price), excludes tickers whose data only starts mid-year (reused-symbol guard), and counts constituents with no data in `StudyResult.coverage`. Sharadar SEP covers delisted names, so remaining `no_data` drops are mostly pre-1998 years (before SEP's history starts) or symbols absent from SEP — don't "clean up" these guards or the coverage report.
- Everything fetched from Sharadar is cached under `data/cache/` (default `data/cache/adj_close_cache.csv`) and served from cache whenever possible. The cache check in `download_adj_close` verifies date coverage (`cache_covers_dates`, 7-day tolerance because the cache index holds trading days only, future `end` clamped to today) and ticker coverage. When dates are covered but tickers are missing, **only the missing tickers are fetched** (over the cache's full span) and merged in; all-NaN columns count as present, so known-empty tickers are never re-requested. When the date range isn't covered, the requested tickers are fully re-downloaded — never splice new date ranges onto cached per-ticker history, because `closeadj` is backward-adjusted and rows fetched at different times sit on different adjustment bases. The cache slice is end-exclusive (`[start, end)`).
- The hypothetical classifier's labeling target in `simulate_custom_portfolio_distribution` is controlled by `label_threshold` (threaded through `run_top_n_study` and `sweep_recall_precision_pairs`): `None` labels stocks against that year's benchmark return; a fixed value (e.g. `0.1`) simulates a model that picks stocks returning more than that absolute threshold. Performance is always *compared* against the actual benchmark — only the classification target changes.
- Multi-year holding horizon: `--horizon H` / `--overlapping` on `study`/`sweep`/`screen` (`horizon`/`overlapping` parameters through `prepare_universe_returns`). Cohorts form at the start of a formation year (membership snapshot at formation, as always) and hold buy-and-hold through year formation+H−1; delistings inside the window keep the last-available-price rule (`year_universe_returns(..., horizon=)`, still counted as `partial_year`). The `stock_yearly`/`bmk_yearly` frames then hold *H-year window totals* indexed by formation year — everything downstream is window-consistent. Non-overlapping windows (default, formation years step by H via `formation_years`) chain into a realizable rebalance-every-H-years path; `--overlapping` gives rolling yearly cohorts (more samples, autocorrelated — the study skips the growth plot, and CAGRs become geometric means of annualized cohort returns, not a path). `label_threshold` and screen criterion values stay **per-annum** and are compounded to window totals internally (`compound_annual_threshold`; `bmk±offset` offsets the benchmark's *annualized* window return) — don't compare annual thresholds to window returns directly. All CAGRs must pass `years_per_period=horizon` to `compute_cagr`. `horizon=1` is exactly the old calendar-year behavior; tests in `tests/test_horizon.py`.
- Plots are written to the repo root: `topNAndCustom_vs_spy.png` and `topNAndCustom_growth.png` from `study`, `recall_precision_heatmap.png` and `recall_precision_heatmap_ew.png` (same grid measured against the equal-weight universe benchmark) from `sweep` (skip with `--no-plots`); PNGs and `data/cache/` are gitignored.
- Two benchmarks wherever performance is reported: the cap-weighted `--benchmark` ticker (SPY) and the equal-weighted universe mean (`equal_weight_benchmark_returns` = `stock_yearly.mean(axis=1)`, bias-consistent with the portfolios). Excess vs SPY mixes selection skill with the equal-weight effect; excess vs the EW benchmark isolates skill. Labeling with `label_threshold=None` still targets the cap-weighted benchmark's return — only the performance comparison gained the second baseline.
- Portfolio aggregation (`portfolio_return`, backing top-N/bottom-N/model-selected returns) accepts an optional ticker-indexed `weights` series for future cap-weighted variants; nothing passes it yet — point-in-time market caps are now obtainable from Sharadar (DAILY/METRICS tables) but not wired in (see TODO.md).
- The sweep computes prices and per-year universe returns once (`prepare_universe_returns`) and re-runs only the classifier simulation per (recall, precision) grid cell (`compute_custom_stats`, looped by `sweep_from_returns`). Don't reintroduce a per-cell `run_top_n_study` call — top-N results don't depend on the grid. Within a cell-year, `simulate_custom_portfolio_distribution` calls `simulate_selection` exactly once (a probe that fixes the achieved metrics, capped (tp, fp), and `num_ways`) and draws the remaining `num_simulations - 1` selections vectorized via `sample_confusion_draws` — don't reintroduce a per-draw `simulate_selection` loop; it's ~300x slower (the old default sweep took ~an hour, now seconds).
- "Miss the super-performers" mode: `exclude_top` (CLI `--exclude-top` on `study`/`sweep`, threaded down to `simulate_custom_portfolio_distribution`) bars the top-k (value >= 1) or top fraction (value in (0,1), `0.1` = top decile) of each year's positives *by return* from the TP pool. The mechanism is the `exclude_names` parameter on `simulate_selection`/`estimate_num_ways` (`simulate_model.py`): excluded names stay in the confusion-matrix denominators — an excluded positive is a forced false negative, so `TP = round(recall*T)` is unchanged and must be filled from ordinary positives — but can't be drawn, and ways-counting uses the shrunken pools. Don't "simplify" by dropping excluded rows from the labeled df; that would silently shrink recall's denominator. Per-year counts surface as `excluded_top` (`custom_stats`, yearly table) and `excluded_top_mean` (summary, sweep rows). When the flag is on, the uniform-draw baseline is computed in the same run with an identically reseeded RNG (so the two distributions differ only by the exclusion): `StudyResult.custom_stats_uniform` (None when off), `*_uniform` + `avg_custom_mean_gap` summary columns, dotted line in both study plots, and per-cell `cagr_custom_q05_uniform`/`custom_q05_gap`/`custom_q05_meets_*_uniform` sweep columns (absent when off — don't rely on a fixed sweep schema across modes).
- Base-rate reporting: `custom_stats` carries a per-year `base_rate` (label prevalence `T/(T+F)` — the precision a random picker gets for free), surfaced as `label_base_rate` in the study's yearly table and as `base_rate_mean/min/max` + `precision_edge_mean` in sweep rows. It depends only on the labeling threshold, never on (recall, precision) — keep it that way, it's the interpretability baseline for the whole study.
- The `screen` subcommand (`screen_label_criteria`/`summarize_screen`) is a *necessary-condition* check: per-year mean return of all stocks meeting a criterion = the precision-1/recall-1 portfolio. Criterion specs: plain numbers are absolute thresholds, `bmk`±offset is benchmark-relative (`parse_criterion`) — the `bmk` threshold is always the cap-weighted benchmark, while excess/verdicts are reported against both benchmarks (`excess`/`passes_screen` vs cap-weighted, `excess_ew`/`passes_screen_ew` vs the equal-weight universe mean). It deliberately runs no selection simulation; don't wire it into the sweep loop.
- Tests live in `tests/` as plain-assert scripts (no pytest dependency), each runnable standalone and none needing network (the Sharadar client is stubbed where relevant): `for t in tests/test_*.py; do uv run python "$t"; done`. A full `pickn.py` run needs a Nasdaq Data Link API key at `~/.nasdaq/data_link_apikey` (AI assistants and CI don't have one) and can take a long time uncached; prefer testing the pure functions (`simulate_model.py`, `year_universe_returns`) when validating changes.

## Environment

- Python >= 3.12, dependencies managed with `uv` (`uv sync`, then `uv run python ...`). See `pyproject.toml`.
- Live Sharadar access needs an API key at `~/.nasdaq/data_link_apikey` (picked up automatically by the `nasdaq-data-link` client). Everything the code fetches lands in `data/cache/` and is reused from there.
